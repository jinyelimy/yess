# yess — Year-End Self System 설계 문서

> **생성**: 2026-04-23 · `/gstack-office-hours` 세션 산출물
> **상태**: DRAFT (v1 착수 전)
> **대상 사용자**: 연말정산 TF 4명 + 유지보수 담당자 약 20명 (총 약 24명, 사내 전용)
> **모드**: Intrapreneurship (사내 도구)

---

## 1. 문제 정의 (Problem Statement)

### 현재 상태 (as-is)
- 연말정산 업무 전반을 **Google Sheets + 공유 드라이브**로 관리 중
- 관리되는 시트 종류
  1. **차수별 패치 리스트** — 회사별 버전/담당자/접속방법/특이사항/반영여부
  2. **후속조치 리스트** — 우선순위, 내용, 처리내역, 담당자, 완료일
  3. **일일 업무 로그** — TF 5명의 개인별 일일 작업내역, 자유 텍스트
  4. **개인 일정** — 연차/반차/출장 등
- 유지보수 담당자 20명이 **패치 반영 여부**를 직접 엑셀에 자유 기재
- TF 4명이 차수별 패치, 후속조치, 일일업무를 각각 별도 시트에 수기로 입력 → 취합/검증 부담

### 핵심 고통 (pain points)
1. **데이터 파편화**: 같은 패치 작업이 "차수별 패치"와 "일일 업무"에 중복 입력됨. 추적 불가.
2. **자유 텍스트 남용**: 담당자·버전·접속방법이 모두 자유 텍스트 → 검색/필터/집계 불가.
3. **진척도 불가시성**: 누가 지금 무엇을 하는지 한눈에 안 보임 → 칸반(Kanban) 니즈.
4. **연도 단위 관리 부재**: 26년/27년 데이터가 섞일 위험, 과거 연도 비교 어려움.
5. **신규 인력 온보딩 어려움**: 회사별 접속 방법·특이사항이 암묵지로만 존재.

### 왜 지금 해결해야 하나
- 연말정산은 **연 단위 반복 업무** → 매년 같은 고통을 다시 겪는 구조.
- 유지보수 담당자 20명 × 관리 회사 다수 → 엑셀 수작업 한계점 도달.
- 26년 현재 진행 중이므로 **27년 시즌 전에 도입하면 다음 시즌이 극적으로 편해짐**.

## 2. 요구사항 (User Requirements)

### 기능 1 — 연말정산 패치 관리 페이지
- 회사별 패치 현황 관리 (버전, 계약형태, 담당자, 접속방법, 반영방법)
- 유지보수 담당자가 **직접 웹에서 패치 반영 여부 기록** (현재는 엑셀)
- 고객사 연락처, 패치 특이사항, 연중 차수별 반영일 관리
- 차수별 패치 리스트 조회 (1차, 2차, 3차...)

### 기능 2 — TF 업무 관리 페이지
- TF 멤버들의 **일일 업무 체계적 관리**
- **칸반 보드** 형식 고려 (스윔레인: 멤버별 또는 상태별)
- 일일 업무 입력 시 **해당 내용이 "차수별 패치"인지 "후속조치"인지 구분** (핵심)
- 개인 일정(연차/반차) 표시

### 비기능 요구사항
- **타임라인**: 3개월 내 최소 기능 프로토타입 배포
- **사용자 규모**: 약 24명 (동시 접속 5~10명)
- **데이터 이관**: 2025~2026년 데이터만 이관 (그 이전은 Google Sheets 잔존)
- **언어**: 한국어
- **운영 환경**: 사내망

## 3. 핵심 전제 (Premises)

| # | 전제 | 근거 |
|---|---|---|
| 1 | 기능1과 기능2는 결국 **같은 데이터 모델**을 공유한다 | 일일업무의 내용이 곧 특정 패치/후속조치이므로 |

| 5 | 실시간 협업(WebSocket)은 v2로 연기, **낙관적 락으로 충분** | 24명이 같은 행을 동시에 수정할 확률 낮음 |

## 5. 데이터 모델

### ER 핵심

```
User (id, name, email, role: TF|MAINTAINER, active)
YearContext (year_cd, start_dt, end_dt, phase)

Company (
  id, year_cd, name,
  version, character_set, java_version,
  contract_type,                    -- 외부/내부
  access_method,                    -- VDI|원격|전달|로컬|VPN
  deploy_method,                    -- O|원격|MAIL|FTP
  year_end_settlement_yn,           -- 연말정산 사용유무
  withholding_tax_yn,               -- 원천세 사용유무
  customer_contacts (jsonb),        -- 고객사 연락처(다수)
  special_notes (text)
)

CompanyMaintainer (company_id, user_id, primary_yn)

Patch (
  id, year_cd, round_no, patch_no,
  company_id,                       -- NULL 가능(공통패치)
  category,                         -- 소스|DB|RD|설정|가이드
  version_scope,                    -- 4.x|5.x|both
  title, special_notes, contents,
  assignee_id,
  status,                           -- DRAFT|IN_PROGRESS|DONE|HOLD
  deploy_date, memo
)

CompanyPatchStatus (                -- 기능1의 핵심 (유지보수자 입력)
  id, company_id, patch_id,
  reflect_yn, reflect_date,
  reflect_method,                   -- 반영|메일|유선
  reflect_note,
  recorded_by (user_id), recorded_at
)

Followup (
  id, year_cd, priority, title, description,
  assignee_id, status,
  start_date, end_date,
  patch_applicable_yn,              -- 패치여부
  result_note
)

DailyLog (
  id, log_date, user_id, content,
  link_type,                        -- PATCH|FOLLOWUP|OTHER
  link_id,                          -- nullable FK
  free_memo
)

PersonalSchedule (
  id, user_id, date_range, type, memo
)
```

### 핵심 통찰 — DailyLog의 다형 참조

`DailyLog.link_type + link_id` 조합이 사용자가 요청한 **"일일 업무 입력 시 차수별 패치인지 후속조치인지 구분"**을 해결한다.

- `link_type=PATCH`, `link_id=Patch.id` → 일일 로그 기록 시 해당 패치의 `status`를 자동 업데이트 힌트
- `link_type=FOLLOWUP`, `link_id=Followup.id` → 후속조치 진척도 가시화
- `link_type=OTHER`, `link_id=NULL` → 회의·교육 등 기타 업무

중복 입력이 사라지고, 패치/후속조치 상세 화면에서 "이 작업에 투입된 일일로그 목록"이 자동 조회된다.

## 6. 아키텍처 대안 분석

### APPROACH A — 최소 MVP (단순 CRUD)
- **범위**: 위 스키마 구현 + 3개 리스트 페이지(일일업무/패치/후속조치)
- **공수**: ~1.5개월 · **Risk**: Low · **Completeness**: 6/10
- **장점**: 빠른 출시, 학습 곡선 제로
- **단점**: 칸반 없음 → TF 만족도 저하, 시각화 부재

### APPROACH B — 통합 모델 + 칸반 + 리스트 뷰 ⭐ **채택**
- **범위**: A + 칸반 보드(dnd-kit) + 리스트↔칸반 뷰 전환 + 일일업무↔패치 링크 UX
- **공수**: ~2.5개월 · **Risk**: Med · **Completeness**: 9/10
- **장점**: 3개월 타임라인에 정확히 부합, TF 칸반 니즈 충족, 기능1 확장 시 스키마 재설계 불필요
- **단점**: 칸반 DnD 구현 공수, 구조화-자유메모 밸런싱 필요

### APPROACH C — 실시간 협업 + 알림 + 히스토리
- **범위**: B + WebSocket + 활동 히스토리 + Slack/Teams 알림 + 파일 첨부 + 템플릿
- **공수**: ~4~5개월 · **Risk**: High · **Completeness**: 10/10 (오버킬)
- **장점**: 완성도 최고
- **단점**: 타임라인 초과, 24명 규모에 과한 복잡도

## 7. 채택된 접근 (Recommended: Approach B)

### 이유
- **타임라인(3개월)과 완성도(9/10)의 교차점**이 정확히 B에 위치
- 스키마가 통합되어 있으므로 **v1은 UI만 기능2에 집중, v1.5에서 유지보수자용 UI 추가**하는 점진적 출시 가능
- 칸반은 TF 4명의 핵심 요구사항이며 B 없이는 엑셀 대비 차별점이 약함

### 출시 단계
| 릴리스 | 범위 | 시점 | 대상 |
|---|---|---|---|
| **v1** | 업무관리 (칸반/리스트) + 패치·후속조치 CRUD | 3개월 | TF 4명 |
| **v1.5** | 유지보수자용 패치 반영 체크 페이지 (기능1 완성) | +1~2개월 | TF + 유지보수 20명 |
| **v2** | 대시보드, 알림, 파일첨부, 연도 롤오버 | 2027년 시즌 전 | 전체 |

## 10. 데이터 이관 전략

### 범위
- **2025년 귀속** 데이터: 패치 리스트, 후속조치, 일일업무 (가능한 범위)
- **2026년 귀속** 데이터: 현재 진행 중인 모든 데이터 (필수)
- 그 이전(24년 이하): Google Sheets에 잔존, 필요 시 수동 조회

### 방식
1. **1회성 Python/Node 스크립트**로 Google Sheets API → PostgreSQL 임포트
2. 컬럼 매핑 규칙 문서화 (자유텍스트 → 구조화 필드 변환 로직)
3. 임포트 후 **TF 2명이 샘플링 검증** (회사 5개, 차수 3개)
4. 검증 후 v1 배포 시점에 실제 데이터 투입

## 11. 미결정 사항 (Open Questions)

1. **인증 방식**: 사내 SSO 유무 확인 필요. (LDAP/OAuth2/자체 JWT/Magic Link 중 선택)
2. **배포 환경**: 사내망 어디에 배포하는가? (사내 Docker 호스트? Kubernetes? VM?)
3. **SMTP 서버**: 알림 이메일 발송을 위한 사내 SMTP 정보
4. **파일 저장소**: 파일 첨부 시 어디에 보관하는가? (S3 호환? 사내 NAS?)
5. **고객사 연락처 다형성**: 첨부사진1의 연락처가 다수(`김지혜, 조민수, ...`)인 경우 많음 → `customer_contacts` jsonb 설계를 어떻게 할지 정의 필요
6. **권한 세분화**: MAINTAINER가 타 회사 데이터 조회 가능 여부 (현재는 자기 담당 회사만 보이는 것을 기본으로 가정)
7. **감사 로그 수준**: 누가 언제 무엇을 수정했는지 이력 추적 범위

## 15. 관찰 (What I noticed)

- 세션 중 사용자가 "프론트 Nest.js"라고 언급 → 기술 스택에 대한 오해가 있었으나 바로 수정 수용. 초기 단계에 이런 오해를 정리하는 것이 v1 착수 전 정말 중요하다.
- 첨부한 Google Sheets 3종(패치 리스트, 일일업무, 후속조치)이 모두 **같은 프로젝트의 다른 뷰**라는 점을 직관적으로 파악하고 있음 → 통합 시스템 설계 의지가 명확.
- 현재 엑셀이 **일정 관리, 패치 추적, 담당자 연락처, 특이사항 문서화**의 4중 역할을 하고 있음 → 이 중 무엇을 대체하고 무엇을 보완할지 v1.5~v2에서 결정 필요.
- "개발자 선호 스택(Nest.js/Postgres)"이 이미 선택되어 있다는 점은 **팀의 기술 성숙도가 일정 수준 이상**임을 시사 → 초기 설계에서 지나치게 단순화하기보다 **확장 가능한 구조**로 설계해도 무리 없을 것.

### 🔴 Critical — 아키텍처의 전제를 흔드는 이슈

#### C1. Build vs Buy 비교 누락 (최우선)
- 이 설계서는 "자체 개발"을 전제했으나, 대안(Notion/Airtable/Linear/Monday.com)과의 비용·기능 비교가 전혀 없음.
- 대략적 비교: SaaS 월 30만원대 vs 개발 인건비 수천만원. 25배 이상 차이.
- **조치**: 사내 보안/규정팀에 "외부 SaaS 사용 가능 여부" 확인 필요. 결과에 따라 설계 방향이 완전히 갈림.
- **상태**: 🟡 OPEN (사내 확인 중) — 프로토타입 착수 전 반드시 해결

#### C2. 인력 리소스 미기재
- 12주 로드맵이 있으나 "몇 명이, 몇 % 가동률로" 작업하는지 없음.
- **상태**: ✅ RESOLVED — **TF 소속 1명 풀타임(1 FTE)**으로 확정

#### C3. "프로토타입" 과 "v1 배포" 혼동
- 프로토타입(내부 시범)과 프로덕션 v1(인증/DR/감사로그 포함)은 공수 차이가 큼.
- **상태**: ✅ RESOLVED — **3개월 목표는 TF 4명 시범용 프로토타입**으로 확정 (v1 프로덕션 승격은 시범 결과에 따라 추후 결정)

### 🟠 High — 설계 결함 (v1 프로덕션 승격 시 반영)

#### H1. DailyLog의 다형성 참조(polymorphic association)는 안티패턴
- 현재 `link_type + link_id`는 FK 제약 불가, Prisma 미지원, JOIN 복잡.
- **상태**: ✅ RESOLVED — **명시적 nullable FK 채택** (`patch_id`, `followup_id`). 프로토타입부터 적용.
- 상세는 [v0-prototype.md §5](./v0-prototype.md) 참고.

#### H2. `Company.year_cd` 잘못된 스키마
- 현재는 "삼양식품"이 26년/27년 **각각 별도 레코드**로 존재하게 됨 → 중복·불일치 위험.
- **상태**: ✅ RESOLVED (프로토타입) / 🟡 DEFERRED (v1.5)
  - 프로토타입: **Company를 영속 실체로, year_cd는 Patch/Followup에만** 부여하는 단순 구조
  - v1.5 이후: 연도별 속성 분리 필요 시 `CompanyYearProfile` 테이블 추가 검토

#### H3. "칸반" 가정 재검증 필요
- TF는 4명. 첨부사진2의 업무 형태는 날짜 × 사람 매트릭스로도 표현 가능.
- **상태**: ✅ RESOLVED — **칸반 보드로 확정** (사용자 결정, 2026-04-23)
  - 프로토타입에서는 "보이는 칸반, 드래그는 v1" 절충안 채택 (dnd-kit 도입은 v1)
